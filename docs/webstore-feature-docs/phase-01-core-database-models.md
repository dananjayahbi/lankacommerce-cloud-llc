**Phase:** 1 of 10  
**Depends on:** Nothing — this is the foundation phase  
**Unlocks:** All other phases  
**Estimated Scope:** Backend only

## 1. Overview

This phase establishes the entire data layer for the webstore feature. The objective is to create a new Django application called `apps.webstore`, define all nine database models, generate the initial migration, wire it into the project's settings and URL configuration, register models in the Django admin, and extend the permissions system to include webstore-specific permission constants.

No business logic, no API views, and no frontend work happens in this phase. The goal is simply to have a clean, working Django app with all tables in the database and a URL skeleton that returns 501 stubs for every planned endpoint.

At the end of this phase, running `python manage.py migrate` must produce all nine new tables with no errors. The Django admin must display all nine models. Every planned API URL must return an HTTP 501 response.

## 2. Existing Codebase Context

> **Read this section carefully before writing a single line of code.** It describes the system this new app integrates with.

### 2.1 Project Structure

All Django apps live under `backend/apps/`. The project is configured in `backend/config/`. Settings are split across `backend/config/settings/base.py` (shared), `development.py`, and `production.py`. The root URL file is `backend/config/urls.py`.

### 2.2 Multi-Tenancy Pattern

Every tenant-scoped model in this codebase has a `tenant` ForeignKey pointing to `apps.tenants.Tenant`. This is the **only** way tenant isolation is enforced at the data layer — there is no row-level security or schema separation. The `Tenant` model lives at `backend/apps/tenants/models.py` and its primary key is a `UUIDField`.

All queries in service layers are expected to filter by `tenant=request.user.tenant`. This is the established convention across all existing apps (`catalog`, `crm`, `pos`, `hr`, etc.).

### 2.3 Existing Models Referenced by Webstore Models

The following existing models are directly referenced (as ForeignKeys) by the new webstore models. Understand their purpose before defining relationships:

| Existing Model | App | Purpose in Webstore |
|---|---|---|
| `Tenant` | `apps.tenants` | Parent scope for all webstore data. Every webstore model has a `tenant` FK. |
| `Plan` | `apps.tenants` | Has a `features: JSONField` column. We add webstore flags to it. |
| `CustomUser` | `apps.accounts` | Staff users (OWNER, MANAGER, etc.). Webstore permissions are added to their `permissions_list`. |
| `Product` | `apps.catalog` | Referenced by webstore collections and orders. Has `tenant` FK and `slug`/handle. |
| `ProductVariant` | `apps.catalog` | Referenced in order line items. Has `sku`, `price`, `stock_quantity`. |
| `Contact` | `apps.crm` | WebstoreCustomer optionally links to a CRM contact via a nullable FK. |
| `Sale` | `apps.pos` | WebstoreOrder links to a POS Sale record once inventory is deducted. |

### 2.4 UUID Primary Keys

Every primary key in this codebase is a `UUIDField(primary_key=True, default=uuid.uuid4, editable=False)`. Follow this convention for all nine new models without exception.

### 2.5 Soft Deletes

Some existing models use a `deleted_at: DateTimeField(null=True, blank=True)` pattern for soft deletes. For webstore models, only use this pattern on `WebstoreOrder` and `WebstoreCustomer`. Other webstore models use hard deletes combined with `is_published` or `is_enabled` boolean flags to represent inactive state.

### 2.6 Settings and INSTALLED_APPS

`INSTALLED_APPS` is defined in `backend/config/settings/base.py`. All existing apps are referenced as `"apps.<appname>"`. The new app must be added as `"apps.webstore"`.

### 2.7 Permissions System

`backend/apps/accounts/constants/permissions.py` defines a dictionary called `ROLE_PERMISSIONS` mapping each role string (e.g., `"OWNER"`, `"MANAGER"`) to a list of permission strings. These permission strings are stored in `CustomUser.permissions_list` (a JSONField) and embedded in JWT tokens at login time. The frontend uses a `can(permission)` helper that checks this array.

### 4.7 `TenantThemeConfig` — A Tenant's Customized Theme Configuration

**Purpose:** This is the most important model in the entire webstore system. It holds the complete visual configuration for a tenant's storefront: global colors, typography, section layouts, and all per-section block settings.

A tenant can have multiple `TenantThemeConfig` records:
- Exactly one with `status = ACTIVE` — this is what consumers see
- Zero or one with `status = DRAFT` — this is what the merchant is currently editing in the customizer

When a merchant publishes, the DRAFT becomes ACTIVE and the old ACTIVE is archived (status set to a third value, or simply deleted — choose consistently).

**Table name:** `webstore_tenant_theme_config`

| Field | Type | Constraints | Description |
|---|---|---|---|
| `id` | UUIDField | PK | |
| `tenant` | ForeignKey(Tenant) | CASCADE, related_name="theme_configs" | |
| `theme` | ForeignKey(WebstoreTheme) | PROTECT, related_name="tenant_configs" | The theme this config is based on. PROTECT prevents deleting a theme that has tenant installs. |
| `status` | CharField(10) | choices=ThemeConfigStatus, default=DRAFT | |
| `config` | JSONField | default=dict | The full theme configuration. See schema below. |
| `purchased_blocks` | JSONField | default=dict | Maps block_type → `{purchased_at, invoice_id}` for premium blocks |
| `created_at` | DateTimeField | auto_now_add | |
| `updated_at` | DateTimeField | auto_now | |
| `published_at` | DateTimeField | null=True, blank=True | Timestamp of last publish action |

**The `config` JSON schema — this is the canonical format used throughout the entire codebase:**

The `config` object has three top-level keys:

1. **`global_settings`** — theme-wide settings that apply across all pages:
   - `colors` object: `primary`, `secondary`, `background`, `text`, `accent` — all HEX strings
   - `typography` object: `heading_font`, `body_font`, `heading_size_scale` (float)
   - `layout` object: `max_content_width` (CSS string), `enable_sticky_header` (boolean)
   - `social` object: `facebook`, `instagram`, `tiktok`, `youtube` — all URL strings or empty

2. **`layout`** — the header and footer configuration (present on all pages). Contains two keys: `header` and `footer`. Each is a section object (see section object format below).

3. **`templates`** — a dictionary keyed by template name (`"index"`, `"product"`, `"collection"`, `"cart"`). Each template value has:
   - `sections` — a dictionary keyed by auto-generated section IDs (UUIDs). Each section object contains:
     - `type` — block type string (e.g., `"hero_banner"`)
     - `disabled` — boolean; false means the section renders
     - `settings` — dictionary of setting ID → value for this section's own settings
     - `blocks` — dictionary of block ID → block object (nested content blocks)
     - `block_order` — ordered array of block IDs (defines render order within the section)
   - `order` — ordered array of section IDs (defines the vertical stack order on the page)

**Meta:** `db_table = "webstore_tenant_theme_config"`

**Application-level constraint (not DB-level):** Enforce in the service layer that at most one ACTIVE and one DRAFT exist per tenant at a time.

- `sku`
- `quantity` — integer
- `unit_price` — decimal
- `total` — decimal (unit_price × quantity)
- `image_url` — snapshot of the product image at time of order
- `requires_shipping` — boolean

**Why denormalized line items?** If a product is later deleted or its price changes, the order must still show the original items and prices. This is standard e-commerce practice.

**Meta:** `db_table = "webstore_order"`, `unique_together = [["tenant", "order_number"]]`

### 7.3 `apps/webstore/urls.py` Structure
