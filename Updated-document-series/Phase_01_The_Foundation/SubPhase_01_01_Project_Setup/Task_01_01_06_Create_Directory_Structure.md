# Task 01.01.06 — Create Directory Structure

## Objective

Create the complete canonical directory structure for both the LankaCommerce Django backend (under `backend/`) and the Next.js frontend (under `frontend/src/` and `frontend/public/`) as defined in `00_Project_Overview.md`, with `.gitkeep` placeholder files ensuring all empty directories are tracked by Git.

## Instructions

### Step 1: Review the Canonical Directory Layout

Before creating any directories, open `document-series/00_Project_Overview.md` and read the directory structure section in its entirety. This section defines the authoritative directory tree for LankaCommerce — every directory listed there must be created in this task. Use this document as the sole source of truth for the project structure. Do not create directories not listed in the canonical layout, and do not omit any that are listed. Deviating from the canonical layout will cause confusion when later tasks reference specific paths.

### Step 2: Create Backend Django Application Directories

Navigate to `backend/apps/`. The `core/` app directory was created in Task 01.01.02 and should already exist. Now create the remaining Django application directories that correspond to the major business domains of LankaCommerce. For each application directory, create the full set of standard Django app files as empty placeholders:

Create `backend/apps/accounts/` — user account management, profiles, and multi-tenancy. Create `backend/apps/catalog/` — product models, categories, and variants. Create `backend/apps/pos/` — point-of-sale transactions and cart operations. Create `backend/apps/crm/` — customer relationship management and supplier management. Create `backend/apps/operations/` — staff, promotions, expenses, and hardware integrations. Create `backend/apps/billing/` — SaaS subscription management and invoicing.

For each of these six application directories (`accounts`, `catalog`, `pos`, `crm`, `operations`, `billing`), create the following placeholder files with empty content: `__init__.py`, `models.py`, `views.py`, `serializers.py`, `urls.py`, `admin.py`, `apps.py`. These placeholder files establish the standard Django REST Framework module structure where each app will eventually define its data models, API views, serializers, URL routing, and admin registrations. The `serializers.py` file is a DRF addition not generated by `startapp` — create it manually in each app directory.

Also create an `__init__.py` at `backend/apps/__init__.py` if it does not already exist, to ensure the `apps` namespace is a proper Python package.

### Step 3: Create Backend Supporting Directories

Inside `backend/`, create the following supporting directories:

- `tests/` — project-level integration and end-to-end tests. Inside `tests/`, create `__init__.py`.
- `docs/` — backend API documentation, architecture diagrams, and developer notes.
- `scripts/` — utility scripts for data migration, maintenance, and DevOps tasks.
- `requirements/` — pip requirements files for environments that prefer them over `pyproject.toml`. Create three files inside: `base.txt`, `development.txt`, and `production.txt`, each as empty files for now.
- `fixtures/` — Django fixture files (JSON or YAML) for seeding reference data. This directory will be populated in Task 01.01.11 and in later sub-phases as models are defined.

### Step 4: Create Frontend Application Route Directories

Navigate to `frontend/src/app/`. The Next.js CLI scaffold created `src/app/layout.tsx` and `src/app/page.tsx`. Now create the route group directories for the three application areas. Note that there is no `src/app/api/` directory in LankaCommerce — all API handling is performed by the Django backend. Do not create an `api/` directory here.

Create the three route group directories: `(auth)/`, `(store)/`, and `(superadmin)/`.

Inside `(auth)/`, create: `login/`, `reset-password/`, and `two-factor/`.

Inside `(store)/`, create: `dashboard/`, `pos/`, `inventory/`, `inventory/[productId]/` (note the square brackets denoting a dynamic route segment), `reports/`, `customers/`, `customers/[customerId]/`, and `settings/`.

Inside `(superadmin)/`, create: `dashboard/`, `stores/`, `users/`, `billing/`, and `settings/`.

These directories represent the full routing surface area for the LankaCommerce frontend — pages created in later sub-phases will land directly inside these directories.

### Step 5: Create Frontend Component Subdirectories

Navigate to `frontend/src/components/`. The `ui/` subdirectory was populated by ShadCN in Task 01.01.04 and should already exist — verify it is present but do not recreate it. Create the following subdirectory hierarchy:

- `pos/` — POS-specific components such as the product search grid, cart line items panel, and payment workflow modal
- `inventory/` — inventory management components such as product creation forms and stock level tables
- `reports/` — chart and data visualisation components used in the reporting sub-phase
- `shared/` — cross-cutting components used across multiple feature areas, such as the navigation sidebar, breadcrumb trail, global data table, skeleton loading states, and confirmation dialogs; inside `shared/`, create a `layout/` subdirectory specifically for the `AppSidebar` and `AppHeader` structural components
- `superadmin/` — super admin-specific components such as the store management table and platform metrics dashboard

### Step 6: Create Frontend Library and Utility Subdirectories

Navigate to `frontend/src/lib/`. The `lib/` directory and a `utils.ts` file were created by the ShadCN CLI in Task 01.01.04 — preserve the existing `utils.ts` file when creating subdirectories. Create the following subdirectories:

- `validators/` — Zod schema files, one per business domain (for example `productSchema.ts`, `customerSchema.ts`, `transactionSchema.ts`)
- `api/` — Django REST API client helper functions. This directory is the frontend's HTTP layer, containing typed fetch wrappers that call the Django backend at `NEXT_PUBLIC_API_BASE_URL`. It replaces the `services/` directory (which was used for Prisma queries in the original VelvetPOS) and the `auth/` directory (which held NextAuth configuration — authentication is now managed entirely by Django)
- `utils/` — general-purpose utility functions that do not belong to a specific domain

Do not create `services/` or `auth/` directories — there are no Prisma queries and no NextAuth configuration in this project. The absence of these directories is intentional and reflects the architecture shift to Django.

### Step 7: Create Hooks, Stores, Types, and Constants Directories

Create `frontend/src/hooks/` to hold all custom React hooks. Create `frontend/src/stores/` to hold the Zustand state management store files — the skeleton files for `cartStore.ts`, `offlineStore.ts`, and `uiStore.ts` will be added here in Task 01.01.10. Create `frontend/src/types/` for TypeScript type definition files and index barrel exports. Create `frontend/src/constants/` for application-wide constant values such as pagination defaults, status enumerations, currency codes, supported locales, and configuration keys that are referenced across multiple modules.

### Step 8: Create Frontend Public Asset Directories

Navigate to `frontend/public/`. Create the following subdirectories: `fonts/` for the self-hosted font files that will be downloaded and placed in Task 01.01.08, `icons/` for SVG icon files and favicon variants, and `images/` for product placeholder images, brand logos, and other static assets.

### Step 9: Add .gitkeep Placeholders to All Empty Directories

Git does not track empty directories. For every directory created in Steps 2 through 8 that does not already contain real files, create an empty file named `.gitkeep` inside it. This includes:

- All `backend/apps/*/` app directories that contain only placeholder Python files (the `.py` files themselves serve as content, so `.gitkeep` is not needed there)
- `backend/tests/`, `backend/docs/`, `backend/scripts/`, `backend/fixtures/`
- All frontend route directories in `src/app/(auth)/`, `(store)/`, and `(superadmin)/` and their sub-routes that do not yet have `layout.tsx` or `page.tsx` files
- All frontend component subdirectories in `src/components/` that have no component files yet
- All `src/lib/` subdirectories (`validators/`, `api/`, `utils/`) that have no files yet
- `src/hooks/`, `src/stores/`, `src/types/`, `src/constants/`
- `frontend/public/fonts/`, `frontend/public/icons/`, `frontend/public/images/`

Directories that should NOT receive `.gitkeep` because they already contain files include: `src/app/` (has `layout.tsx` and `page.tsx`), `src/components/ui/` (has ShadCN component files), `src/lib/` (has `utils.ts` from ShadCN), and all `backend/apps/*/` directories that already have their placeholder Python files.

### Step 10: Verify Both Directory Trees

From inside `backend/`, run `find . -type d | sort` to list all directories. From inside `frontend/`, run `find src -type d | sort` and `find public -type d | sort`. Compare each output against the canonical layout in `00_Project_Overview.md`. If any directory is missing, create it and add `.gitkeep`. If any extra directory exists that is not in the canonical layout, remove it. Only proceed when both directory trees match the canonical layout exactly.

## Expected Output

- All canonical backend directories from `backend/apps/` to `backend/fixtures/` exist, with each Django app having its standard placeholder Python files
- All canonical frontend directories from `frontend/src/app/` route groups through `frontend/src/constants/` and `frontend/public/` asset directories exist
- Every empty directory contains a `.gitkeep` file so that it is committed to source control
- No `src/app/api/` directory exists in the frontend (Django handles all API endpoints)
- No `src/lib/services/` or `src/lib/auth/` directory exists (these patterns belong to the old Next.js monolith architecture)
- `src/lib/api/` exists as the frontend HTTP client layer for Django REST calls
- `git status` shows all new `.gitkeep` files as untracked additions awaiting staging

## Validation

- [ ] `find backend -type d | sort` output matches the canonical backend structure from `00_Project_Overview.md`
- [ ] `find frontend/src -type d | sort` output matches the canonical frontend structure
- [ ] Every empty directory contains a `.gitkeep` file
- [ ] `src/app/api/` does NOT exist in the frontend directory tree
- [ ] `src/lib/api/` DOES exist in the frontend directory tree
- [ ] `src/components/ui/` contains the ShadCN component files from Task 01.01.04 and no `.gitkeep`
- [ ] `src/lib/utils.ts` is preserved and not overwritten

## Notes

The `src/lib/api/` directory introduced in this task is a critical architectural distinction between LankaCommerce and the original VelvetPOS. In VelvetPOS, the `services/` directory held Prisma ORM queries that ran server-side in Next.js. In LankaCommerce, all data access goes through HTTP requests to the Django REST API. The `api/` directory will hold typed fetch helper functions and Axios/fetch wrappers that make those HTTP calls, so all frontend data dependencies are clearly channelled through this single directory. When a developer wants to understand how the frontend fetches data, they look in `src/lib/api/` — this creates a consistent, auditable boundary between the frontend and the backend.
