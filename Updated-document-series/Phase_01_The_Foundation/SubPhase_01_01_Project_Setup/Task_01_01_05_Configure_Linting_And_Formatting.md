## Objective

Configure the full code quality toolchain for both the Django backend and the Next.js frontend. The frontend receives ESLint with TypeScript strict rules, Prettier with project-wide formatting standards, and Husky lint-staged pre-commit hooks. The backend receives Ruff for Python linting, Black for Python formatting, and isort for import ordering — all configured in `pyproject.toml` with Makefile targets for easy invocation.

## Instructions

### Step 1: Install Frontend ESLint Plugins and Dependencies

All frontend steps are run from inside the `frontend/` directory. The Next.js CLI scaffold already installed `eslint` and `eslint-config-next` during Task 01.01.01. Extend that baseline by installing the TypeScript ESLint toolchain:

```
pnpm add -D @typescript-eslint/parser @typescript-eslint/eslint-plugin
```

These two packages provide TypeScript-aware linting rules that analyse type information beyond what Next.js's default ESLint configuration covers. Also install the import ordering plugin:

```
pnpm add -D eslint-plugin-import
```

This plugin enforces import ordering and grouping, keeping the codebase consistent in how dependencies are imported across all files.

### Step 2: Configure the Frontend ESLint Flat Config

Open `eslint.config.mjs` (inside `frontend/`). The Next.js CLI may have placed a minimal configuration here. Replace the entire file content with a properly structured flat configuration array. The array should include the Next.js core web vitals configuration imported from `"eslint-config-next/core-web-vitals"`, a TypeScript ESLint configuration object applying to all `.ts` and `.tsx` files that includes the `"@typescript-eslint/recommended-type-checked"` rule preset with the `languageOptions.parserOptions` specifying `project` set to `"./tsconfig.json"` to enable type-aware linting, and a custom rules override object. In the rules override, set `"@typescript-eslint/no-explicit-any"` to `"error"` so any use of TypeScript's `any` type fails the lint check as an error rather than a warning. Set `"@typescript-eslint/no-unused-vars"` to `"error"` with options to check all variables and caught errors. Set `"no-console"` to `"warn"` so that `console.log` or `console.error` statements are flagged and must be addressed before production. Set `"react/self-closing-comp"` to `"error"` to enforce self-closing JSX tags on components with no children. Configure `"import/order"` to `"warn"` with groups ordering imports as `builtin`, `external`, `internal`, `parent`, `sibling`, and then `index`, with blank lines between each group.

### Step 3: Install and Configure Frontend Prettier

From inside `frontend/`, run:

```
pnpm add -D prettier prettier-plugin-tailwindcss
```

Create a file named `prettier.config.mjs` in the `frontend/` directory. The file should export a default configuration object with the following settings: `printWidth` set to `100`, `tabWidth` set to `2`, `singleQuote` set to `true`, `trailingComma` set to `"all"`, `semi` set to `true`, and `plugins` set to an array containing the string `"prettier-plugin-tailwindcss"`. Add a `tailwindConfig` key pointing to `"./tailwind.config.ts"` so the Tailwind plugin can read the LankaCommerce configuration and sort classes in the project's canonical token order.

### Step 4: Add Frontend Scripts to package.json

Open `frontend/package.json` and add the following entries to the `"scripts"` object:

- `"lint"`: `"eslint src/ --max-warnings=0"` — the `--max-warnings` flag causes ESLint to exit with a non-zero code if any warnings are present, treating warnings as errors in CI contexts
- `"lint:fix"`: `"eslint src/ --fix"` — for auto-correctable issues
- `"format"`: `"prettier --write ."` — format all files in the frontend project
- `"format:check"`: `"prettier --check ."` — verify formatting without making changes, useful in CI pipelines

### Step 5: Install and Initialise Husky

From inside `frontend/`, run:

```
pnpm add -D husky lint-staged
```

Then run:

```
pnpm exec husky init
```

This creates a `.husky/` directory inside `frontend/` and automatically adds a `"prepare"` script to `package.json` that runs `"husky"` on every invocation of `pnpm install`. The prepare hook ensures every developer who clones the repository and runs `pnpm install` inside `frontend/` gets Husky hooks activated automatically without any manual configuration step.

### Step 6: Configure the Pre-Commit Hook

Open the `.husky/pre-commit` file generated by Husky init. Replace its default placeholder content with a single line:

```
pnpm exec lint-staged
```

This delegates all pre-commit processing to lint-staged, which is configured in the next step. Keeping the pre-commit hook minimal and delegating to lint-staged is the recommended practice because it allows lint-staged config to be version-controlled in `package.json` and easily updated without touching the hook file.

### Step 7: Configure lint-staged in package.json

Open `frontend/package.json` and add a top-level `"lint-staged"` key. Configure rules for two file type patterns: files matching `"**/*.{ts,tsx}"` should be processed by `"eslint --fix"` followed by `"prettier --write"` — in that order, because ESLint's auto-fix may change code that Prettier then needs to format; files matching `"**/*.{json,md,css}"` should be processed only by `"prettier --write"`. This targeted approach runs linting only on staged files, keeping pre-commit execution time fast as the project grows.

### Step 8: Install Backend Python Linting Tools

All backend steps are run from inside the `backend/` directory. Install Ruff, Black, and isort as development-only dependencies using Poetry's `--group dev` flag:

```
poetry add --group dev ruff black isort
```

This installs them into a `[tool.poetry.group.dev.dependencies]` section in `pyproject.toml`, keeping them separate from production dependencies. They will not be installed when a deployment environment runs `poetry install --no-dev`.

### Step 9: Configure Backend Linting Tools in pyproject.toml

Open `backend/pyproject.toml`. Add the following configuration sections after the existing `[tool.poetry]` and `[tool.poetry.dependencies]` blocks:

```toml
[tool.ruff]
line-length = 100
target-version = "py312"
select = ["E", "F", "W", "I"]

[tool.ruff.lint]
select = ["E", "F", "W", "I"]
ignore = []

[tool.black]
line-length = 100
target-version = ["py312"]

[tool.isort]
profile = "black"
line_length = 100
```

The `select` array for Ruff enables: `E` (pycodestyle error rules), `F` (Pyflakes rules for undefined names and unused imports), `W` (pycodestyle warning rules), and `I` (isort import ordering rules via Ruff's built-in isort integration). The `line-length` of 100 is shared across all three tools to ensure consistent line wrapping behaviour. The `"black"` profile for isort makes its output compatible with Black's formatting decisions, preventing the two tools from conflicting on import block layout.

### Step 10: Add Backend Makefile Targets for Linting

Open `backend/Makefile`. The `lint`, `format`, and `format-check` targets were stubbed out in Task 01.01.02. Verify they now correctly invoke the tools that have been installed in this task:

```makefile
lint:
	poetry run ruff check .

format:
	poetry run black . && poetry run isort .

format-check:
	poetry run black --check . && poetry run isort --check-only .
```

If the targets were not added in Task 01.01.02, add them now. These targets give developers a single consistent command to run quality checks regardless of which tools are configured behind the scenes.

### Step 11: Verify the Full Toolchain

From inside `frontend/`, run `pnpm lint` and confirm it exits with zero warnings and zero errors on the current skeleton codebase. Run `pnpm format:check` and confirm Prettier reports no formatting violations. From inside `backend/`, run `poetry run ruff check .` and confirm it exits cleanly. Run `poetry run black --check .` and confirm no formatting violations are reported. Stage a test file in the frontend with intentional style violations (such as double quotes or a missing trailing comma) and attempt a git commit — verify the pre-commit hook triggers, lint-staged runs, and the violations are either auto-fixed or the commit is blocked.

## Expected Output

- `frontend/eslint.config.mjs` enforces TypeScript strict rules, `no-console` warnings, `no-explicit-any` errors, and import ordering
- `frontend/prettier.config.mjs` enforces 2-space indent, single quotes, trailing commas, 100-character line limit, and Tailwind class sorting
- `frontend/.husky/pre-commit` is present, executable, and delegates to lint-staged
- `frontend/package.json` contains lint-staged configuration and four new scripts: `lint`, `lint:fix`, `format`, `format:check`
- `backend/pyproject.toml` contains `[tool.ruff]`, `[tool.black]`, and `[tool.isort]` configuration sections
- `pnpm lint` and `pnpm format:check` both exit cleanly on the frontend skeleton
- `poetry run ruff check .` exits cleanly on the backend

## Notes

The `"prepare"` script that Husky adds to `frontend/package.json` is the mechanism that installs Git hooks automatically for all contributors. If a contributor's environment runs `pnpm install` with the `--ignore-scripts` flag (common in some CI environments), the prepare script will not run and hooks will not be activated. For CI pipelines, linting and formatting should be run explicitly as separate pipeline steps rather than relying on pre-commit hooks, which are a developer convenience rather than a CI enforcement mechanism. On the backend, Ruff is the primary linting tool and also handles import ordering via its built-in isort integration (`I` rule group). The standalone `isort` package is included for compatibility and for the `make format` target — some editors and IDE integrations expect `isort` to be a directly invocable command rather than going through Ruff.
