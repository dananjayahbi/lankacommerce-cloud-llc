# Task 01.01.01 — Initialize Monorepo Projects

## Metadata

| Field | Value |
|---|---|
| Sub-Phase | SubPhase 01.01 — Project Setup & Configuration |
| Phase | Phase 01 — The Foundation |
| Estimated Complexity | Medium |
| Dependencies | None |

## Objective

Bootstrap the LankaCommerce monorepo by creating both the Django REST Framework backend and the Next.js 15+ frontend as sub-projects within a single Git repository, producing a clean project skeleton ready for database, design system, and tooling configuration in subsequent tasks.

## Instructions

### Step 1: Verify Prerequisites

Before creating the project, confirm that the development environment has all required tools installed. Verify that Node.js version 20 or later is active by running `node --version` in the terminal. Confirm that pnpm version 9 or later is installed — if it is not present, install it globally by running `npm install -g pnpm`. Verify that Python 3.12 or later is available by running `python --version` or `python3 --version`. Confirm that Poetry is installed by running `poetry --version` — if it is absent, install it using the official installer at `https://python-poetry.org/docs/#installation` (the recommended method is `pipx install poetry` or the standalone installer script). Confirm that a PostgreSQL server (either local or cloud-managed) is accessible and that credentials are at hand, as they will be required in Task 01.01.02. Having all prerequisites confirmed before beginning avoids interruptions later in the setup flow.

### Step 2: Create the Monorepo Root

Navigate in your terminal to the parent directory where the LankaCommerce repository should live. Create the root project directory by running `mkdir lanka-commerce` and then change into it with `cd lanka-commerce`. Initialise a Git repository by running `git init`. Create a root-level `.gitignore` file that covers both Python and Node.js artefacts. The file should include the following entries at minimum: `__pycache__/`, `*.py[cod]`, `*.egg-info/`, `.venv/`, `dist/`, `build/`, `.env`, `*.env.local`, `*.env.*.local`, `node_modules/`, `.next/`, `out/`, `*.tsbuildinfo`, `.DS_Store`, `Thumbs.db`, `.idea/`, `.vscode/`. This single root `.gitignore` covers both sub-projects and avoids the need for separate ignore files in each sub-directory, while still allowing per-sub-project additions for more specific rules.

### Step 3: Initialize the Django Backend

From the monorepo root, create the backend directory by running `mkdir backend`. Change into it with `cd backend`. Initialise a new Poetry project by running `poetry init`. When the interactive CLI prompts for the package name, enter `lanka-commerce-backend`. When prompted for a version, accept the default `0.1.0`. When prompted for a description, enter `LankaCommerce SaaS ERP — Django REST Framework backend`. When prompted for an author, enter your name and email. When prompted for Python version compatibility, enter `^3.12`. Decline the interactive dependency addition prompts (you will add dependencies explicitly in the next sub-step) by answering `no` to both the main dependencies and development dependencies prompts.

After `poetry init` completes and creates `pyproject.toml`, add the core production dependencies in a single command:

```
poetry add django djangorestframework djangorestframework-simplejwt django-cors-headers psycopg2-binary python-decouple
```

This installs Django 4.2+ or 5.x (whichever Poetry resolves), Django REST Framework for the API layer, SimpleJWT for token-based authentication (configured in SubPhase 01.02), CORS headers middleware for cross-origin requests from the Next.js frontend, the psycopg2-binary PostgreSQL adapter, and python-decouple for reading environment variables from `.env` files in a clean, type-safe manner.

Once the packages are installed, create the Django project structure inside the `backend/` directory by running:

```
poetry run django-admin startproject config .
```

The trailing dot (`.`) is essential — it instructs `django-admin` to create the project in the current directory rather than creating an additional nested subdirectory. After this command, the `backend/` directory should contain `manage.py` and a `config/` directory with `settings.py`, `urls.py`, `wsgi.py`, and `asgi.py`.

### Step 4: Create the Django Settings Structure

The default `config/settings.py` file generated by `django-admin` is a monolithic single file. Split it into a layered settings structure. Create a `settings/` sub-directory inside `config/` by running `mkdir config/settings`. Move the generated `settings.py` into this new directory and rename it to `base.py` by running `mv config/settings.py config/settings/base.py`. Create an empty `config/settings/__init__.py` file to make the directory a Python package. Create `config/settings/development.py` with the following content:

```python
from .base import *  # noqa: F401, F403

DEBUG = True

ALLOWED_HOSTS = ['localhost', '127.0.0.1']
```

Create `config/settings/production.py` with the following skeleton content:

```python
from .base import *  # noqa: F401, F403

DEBUG = False

# ALLOWED_HOSTS and other production settings will be configured during deployment.
```

Open `manage.py` and update the `DJANGO_SETTINGS_MODULE` value from `config.settings` to `config.settings.development`. Similarly, open `config/wsgi.py` and `config/asgi.py` and update the `DJANGO_SETTINGS_MODULE` default from `config.settings` to `config.settings.development`. This ensures the development settings are the default when running locally, while the production settings module can be activated by setting the `DJANGO_SETTINGS_MODULE` environment variable in the deployment environment.

### Step 5: Initialize the Next.js Frontend

Return to the monorepo root by running `cd ..` from inside `backend/`. Create the frontend directory by running `mkdir frontend`. Change into it with `cd frontend`. Run the Next.js project initialiser by executing:

```
pnpm create next-app@latest .
```

The trailing dot (`.`) initialises the project in the current directory rather than creating an additional nested folder. The interactive CLI will present a series of prompts; answer each one as follows: accept TypeScript (`yes`), accept ESLint (`yes`), accept Tailwind CSS (`yes`), accept the `src/` directory convention (`yes`), accept the App Router (`yes`), decline the Turbopack option for now (`no` — Turbopack will be evaluated once stability in the project context is confirmed), and set the import alias to `@/*` mapping to `./src/*`. Review each prompt answer before pressing Enter to avoid misconfigurations that would require recreating the project.

After the CLI completes, confirm the following files and directories were scaffolded: `src/app/layout.tsx`, `src/app/page.tsx`, `src/app/globals.css`, `public/`, `tailwind.config.ts`, `tsconfig.json`, `next.config.ts`, `package.json`, and `eslint.config.mjs`. If any of these are missing, the project should be recreated from a clean state.

### Step 6: Configure pnpm as the Enforced Package Manager

Change into the `frontend/` directory. Open `package.json` and add a `"packageManager"` field set to the exact pnpm version string currently installed on your system, for example `"pnpm@9.x.x"`. This field is read by Corepack and Node.js tooling to enforce consistent package manager usage across all contributors. Also create a file named `.npmrc` in the `frontend/` directory and add the line `engine-strict=true` to it. This ensures that contributors using the wrong Node.js or pnpm version receive an immediate warning rather than a silent failure. Run `pnpm install` inside `frontend/` to generate `pnpm-lock.yaml`. Confirm that `node_modules/` is populated and `pnpm-lock.yaml` is present. If a `package-lock.json` or `yarn.lock` file was accidentally created at any point, delete it and rerun `pnpm install`.

### Step 7: Add Root-Level Tooling Scripts

Return to the monorepo root (`cd ..` from `frontend/`). Create a root-level `package.json` with the following content to provide convenient scripts for running both services from the repository root:

```json
{
  "name": "lanka-commerce",
  "version": "0.1.0",
  "private": true,
  "scripts": {
    "dev:backend": "cd backend && poetry run python manage.py runserver",
    "dev:frontend": "cd frontend && pnpm dev",
    "lint:frontend": "cd frontend && pnpm lint",
    "format:frontend": "cd frontend && pnpm format:check",
    "lint:backend": "cd backend && poetry run ruff check .",
    "format:backend": "cd backend && poetry run black --check ."
  }
}
```

This root `package.json` is not a Node.js workspaces configuration — it is purely a convenience script runner. The backend and frontend remain independent projects with their own dependency management. Note that the root `package.json` should be committed to source control as part of the repository scaffolding.

### Step 8: Verify Both Servers Start

Open two terminal windows. In the first, navigate to the monorepo root and run `cd backend && poetry run python manage.py runserver`. Confirm the Django development server starts and the terminal displays the `Starting development server at http://127.0.0.1:8000/` message without errors. Open a browser tab and navigate to `http://localhost:8000` — Django should display its default "The install worked successfully! Congratulations!" page. In the second terminal, run `cd frontend && pnpm dev`. Confirm the Next.js development server starts on port 3000. Navigate to `http://localhost:3000` and confirm the default Next.js welcome page loads. Stop both servers with `Ctrl+C` once successful loading is confirmed.

### Step 9: Remove Next.js Boilerplate

Change into `frontend/`. Open `frontend/src/app/page.tsx` and replace the entire contents with a minimal React functional component:

```tsx
export default function HomePage() {
  return (
    <main>
      <h1>LankaCommerce — Development In Progress</h1>
    </main>
  );
}
```

Open `frontend/src/app/globals.css` and remove all default Tailwind `@layer base` overrides and custom CSS that the Next.js CLI injected. Retain only the three Tailwind directives (`@tailwind base`, `@tailwind components`, `@tailwind utilities`). The full design token variable declarations will be added in Task 01.01.03.

## Expected Output

At the end of this task, the repository should contain:

- A monorepo root directory `lanka-commerce/` with a `.gitignore` covering Python and Node.js artefacts and a root-level `package.json` with convenience scripts
- A `backend/` directory with a fully initialised Django project using the layered settings structure (`config/settings/base.py`, `development.py`, `production.py`) and all production dependencies installed via Poetry
- A `frontend/` directory with a fully scaffolded Next.js 15+ project using TypeScript, the App Router, Tailwind CSS, and the `src/` directory convention, with `pnpm-lock.yaml` present and no other lock files
- A valid `.npmrc` in `frontend/` with `engine-strict=true`
- A clean `frontend/src/app/page.tsx` containing only the LankaCommerce development placeholder
- A clean `frontend/src/app/globals.css` containing only the three Tailwind CSS directives
- Both the Django server (port 8000) and the Next.js server (port 3000) start cleanly

## Validation

- [ ] `poetry run python manage.py runserver` starts Django on port 8000 without errors
- [ ] `pnpm dev` (inside `frontend/`) starts Next.js on port 3000 without errors
- [ ] `pnpm-lock.yaml` is present in `frontend/` and no other lock files exist
- [ ] The `"packageManager"` field in `frontend/package.json` references pnpm with a version string
- [ ] The `@/*` import alias is configured in `frontend/tsconfig.json` pointing to `./src/*`
- [ ] `backend/config/settings/development.py` exists and `manage.py` references `config.settings.development`
- [ ] `frontend/src/app/page.tsx` contains only the minimal LankaCommerce placeholder content
- [ ] `frontend/src/app/globals.css` contains only the three Tailwind directives

## Notes

The Django project is initialised inside `backend/` with the dot-form of `startproject`, meaning `manage.py` lives at `backend/manage.py` and the config package lives at `backend/config/`. All `manage.py` and `poetry run` commands in subsequent tasks are run from inside the `backend/` directory unless stated otherwise. Similarly, all `pnpm` commands in subsequent tasks are run from inside the `frontend/` directory unless stated otherwise. Keeping these two working directories distinct is important for avoiding confusion in multi-window terminal workflows.
