# Task 02.01.04 — Create StockMovement Model

## Metadata

| Property | Value |

| Estimated Complexity | Medium |

---

## Objective

This task defines the `StockMovement` model, which serves as an immutable audit log for all changes to a product variant's stock quantity. Every time inventory changes — whether through a sale, a purchase order receipt, a manual correction, or a stock-take reconciliation — a corresponding `StockMovement` record is created and never modified. The model is intentionally append-only and carries no `updated_at` or `deleted_at` fields.

---

## Instructions

### Step 1: Define the StockMovementReason Enumeration

In `backend/apps/catalog/models.py`, define a `StockMovementReason` class inheriting from `django.db.models.TextChoices`. This enumeration must declare the following nine choices:

- `SALE` — stock reduced by a point-of-sale transaction
- `SALE_RETURN` — stock restored by a return or exchange at the POS
- `PURCHASE_RECEIPT` — stock increased by receiving goods against a supplier purchase order
- `MANUAL_ADJUSTMENT` — a direct stock correction performed by a staff member
- `STOCK_TAKE_ADJUSTMENT` — an automated adjustment generated by a completed and approved stock-take session
- `DAMAGE_WRITE_OFF` — stock removed due to items that are damaged, expired, or unsellable
- `TRANSFER_IN` — stock received from another location or warehouse
- `TRANSFER_OUT` — stock dispatched to another location or warehouse
- `INITIAL_STOCK` — the opening inventory entry created when a variant is first added to the system

### Step 2: Define the StockMovement Model

Define the `StockMovement` model inheriting from `django.db.models.Model`. Add the following fields:

- **id**: A `UUIDField` primary key with `uuid.uuid4` default and `editable=False`.
- **tenant**: A `ForeignKey` to the `Tenant` model with `on_delete=CASCADE`, `db_index=True`, and `related_name='stock_movements'`.
- **variant**: A `ForeignKey` to the `ProductVariant` model with `on_delete=PROTECT` and `related_name='movements'`. The `PROTECT` deletion rule is deliberate: a variant that has stock movement history cannot be hard-deleted, preserving the audit trail.
- **reason**: A `CharField` with `choices=StockMovementReason.choices` and `max_length=30`.
- **quantity_delta**: An `IntegerField`. Positive values indicate stock added to the variant; negative values indicate stock removed.
- **quantity_before**: An `IntegerField`. Records the variant's `stock_quantity` immediately before this movement was applied.
- **quantity_after**: An `IntegerField`. Records the variant's `stock_quantity` immediately after this movement was applied.
- **actor**: A `ForeignKey` to the `CustomUser` model from `apps.accounts` with `on_delete=SET_NULL`, `null=True`, and `related_name='stock_movements'`. Records the user who performed or triggered this inventory change. Using `SET_NULL` means that deleting a user account does not cascade to delete audit records.
- **note**: An optional `TextField` with `blank=True` and `null=True`. Allows staff to provide a free-text explanation, particularly useful for manual adjustments.
- **sale_id**: An optional `UUIDField` with `null=True` and `blank=True`. Stores the ID of the related sale transaction without a formal database ForeignKey constraint. The Sale model does not yet exist at this stage of the build, so a plain UUID field is used to avoid a circular dependency.
- **purchase_order_id**: An optional `UUIDField` with `null=True` and `blank=True`. Follows the same loose-reference pattern as `sale_id` for purchase order associations.
- **stock_take_session_id**: An optional `UUIDField` with `null=True` and `blank=True`. References the stock-take session that triggered this adjustment, if applicable.
- **created_at**: A `DateTimeField` with `auto_now_add=True`. This is the only timestamp on the model.

`StockMovement` deliberately omits both `updated_at` and `deleted_at`. Movements are permanent, immutable audit records. Once written, they must never be modified or removed.

### Step 3: Define Indexes

In the `StockMovement` model's inner `Meta` class, define the following `models.Index` entries:

- A composite index on `['variant', 'created_at']` for efficient chronological movement history queries per variant.
- A composite index on `['tenant', 'created_at']` for efficient cross-variant reporting queries scoped to a single tenant.
- A single-column index on `['actor']` for lookups of all movements performed by a specific user.

Set `verbose_name` and `verbose_name_plural` in `Meta`. Add `ordering = ['-created_at']` so that movements are returned in reverse chronological order by default without requiring an explicit `order_by` call in every query.

Add a `__str__` method that returns a human-readable string combining the reason, the `quantity_delta`, and the variant SKU.

### Step 4: Register StockMovement in Django Admin

Open `backend/apps/catalog/admin.py`, import `StockMovement`, and register it with `admin.site.register`. Define a `StockMovementAdmin` class that sets `list_display` to include `reason`, `variant`, `quantity_delta`, `quantity_before`, `quantity_after`, `actor`, and `created_at`. Set `readonly_fields` to include all fields on the model — this reinforces the immutability of movement records within the admin UI. Add `list_filter` for `reason` and `tenant`.

### Step 5: Generate and Apply the Migration

Run `makemigrations catalog --name add_stock_movement_model` via `poetry run` to generate the migration file. Then run `migrate` to apply it to the database.

## Notes

- The `on_delete=PROTECT` constraint on the `variant` ForeignKey is a critical business rule. Stock movement history must be preserved even if a variant is soft-deleted. Hard deletion of a variant is blocked at the database level by this constraint.
- The loose UUID references (`sale_id`, `purchase_order_id`, `stock_take_session_id`) follow the same pattern used across the LankaCommerce architecture for cross-app references at early build stages. Application-level consistency between these IDs and the referenced records is enforced in the service layer, not at the database layer.
- The `actor` field using `SET_NULL` means that historical stock movements remain intact after a staff member's user account is deleted. A null actor in an old movement indicates that the originating user account no longer exists.
